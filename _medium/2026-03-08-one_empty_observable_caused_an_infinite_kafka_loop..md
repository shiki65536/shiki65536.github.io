Last week I investigated a production incident involving a Kafka consumer.The symptom looked like a Kafka issue at first: the consumer appeared stuck and new messages weren’t being processed.But the real issue was much smaller, and much sneakier.The SymptomLogs showed this error: EmptyError: no elements in sequenceThe consumer kept retrying the same offset message. The service processed the message, failed, restarted, and then processed the same offset again. Over and over.Why the Consumer Got “Stuck”Kafka works with at-least-once delivery in default. A message offset is committed only after processing succeeds.So the actual loop looked like this:Kafka sends message (offset 1234) ↓Application processes message ↓Runtime error occurs ↓Offset never commits ↓Consumer restarts ↓Kafka sends offset 1234 againFrom the outside it looks like Kafka is stuck. But Kafka is actually doing exactly what it promises.The Actual BugThe root cause turned out to be a small RxJS detail. Inside an error handler:catchError((err) =&gt; { logError(err) return of()})At first glance this looks harmless.But of() without arguments creates an Observable that emits nothing and completes immediately. Later in the pipeline, some operators like lastValueFrom, combineLatest expect at least one value.Since nothing was emitted, RxJS threw EmptyError. That exception bubbled up into the Kafka message handler. And once the handler throws, the offset never commits.The FixInstead of returning an empty Observable, the handler now returns a default value.This guarantees the Observable emits at least one value, allowing the pipeline to complete normally.The Kafka handler finishes, and the offset commits.Final ThoughtDistributed systems often fail in dramatic ways.But the root cause is frequently small.A missing value.An unchecked assumption.Or in this case, an Observable that emitted nothing at all.
