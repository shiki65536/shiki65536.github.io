Recently we ran into an incident where one of our services suddenly stopped sending logs to OpenSearch and New Relic.At first glance it looked like an observability issue. But the service itself was still healthy.The timeline was roughly:Before End of Feb: dashboards updating normallyEarly March: no new logs appeared in OpenSearch and NewRelicMid Marxh: we finally noticed the gapThe strange part was that logs still existed inside AWS.The Logging PipelineThe pipeline looked roughly like this:Lambda → Kafka (MSK proxy) → downstream pipeline → OpenSearch / NewRelic / dashboardsThe Lambda collects logs and publishes them to Kafka. From there they flow through consumers and eventually land in OpenSearch.If Kafka ingestion breaks, the dashboards simply stop updating.The ClueIn the Lambda logs we saw errors like this:DNS lookup failed for b1.org-msk-prod….kafka.ap-southeast-2.amazonaws.com:9096At first glance this looked like a DNS problem.But the real issue turned out to be network reachability.The Root CauseA change was made in shared services to add a new VPC endpoint. Along with that endpoint, a new security group was introduced.However, the new SG did not whitelist traffic from the Lambda that writes logs to Kafka. As a result:Lambda still executedCloudWatch logs still existedbut the Lambda could not connect to KafkaNo Kafka writes → no downstream processing → no logs in OpenSearch.Once the security group rule was fixed and the service redeployed, logs immediately started flowing again.Why This Was TrickyNothing in the application looked broken.The service was healthyLambda was runningCloudWatch logs existedno user-facing errors occurredThe failure happened in the network boundary of the observability pipeline.Which meant our monitoring system itself became partially blind.LessonsA few takeaways from this incident:Observability pipelines depend on network connectivity.Logging systems are still distributed systems. They rely on DNS, routing, security groups, and endpoints.CloudWatch logs existing does not guarantee end-to-end observabilityInternal logs can exist while external dashboards remain empty.Alert on missing dataWe’re now considering alerts for:- sudden drops in ingestion rate- dashboards that stop receiving new data- repeated connectivity failures in log pipelinesThe service never stopped producing logs.We simply lost the path that delivered them.
